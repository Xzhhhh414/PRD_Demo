# run-tests — 测试执行工作流

输入参数：$ARGUMENTS（可选，指定要测试的文件/模块/功能范围）

---

## 执行流程

### Step 1：分析项目测试环境

扫描项目结构，识别：
- 测试框架：Jest / Vitest / Mocha / pytest / 其他
- 测试目录：`__tests__/`、`*.test.*`、`*.spec.*`、`tests/`
- 测试配置：`jest.config.*`、`vitest.config.*`、`pytest.ini` 等
- 测试脚本：`package.json` 中的 `test` / `test:*` 命令

如果项目没有配置测试框架，询问用户：
```
项目尚未配置测试框架。建议根据项目技术栈选择：
1. Vitest（推荐，Vite 项目）
2. Jest（通用）
3. 其他（请指定）

请选择，或输入 "跳过" 不配置：
```

### Step 2：确定测试范围

**如果指定了 `$ARGUMENTS`**：
- 文件路径 → 只测试该文件对应的功能
- 模块/功能名 → 搜索相关文件

**如果没有指定**：
- 使用 `git diff` 获取最近修改的文件
- 或扫描 `dev-log/` 最新的开发文档，获取变更文件清单

### Step 3：生成/更新测试

调用 Test Runner Agent：

```
subagent_type: "prd-workflow:Test Runner"
prompt: |
  请为以下文件/功能生成测试用例。

  ## 测试范围
  {Step 2 确定的文件列表}

  ## 需求文档
  {如果有关联的 PRD 或 dev-log，粘贴相关内容}

  请：
  1. 阅读被测文件的实现代码
  2. 从需求文档提取预期行为
  3. 生成测试文件（正向 + 反向 + 边界）
  4. 按项目现有测试框架和风格编写
```

### Step 4：执行测试

运行项目的测试命令：
- 如果有 `package.json`，使用 `npm test` 或对应的测试脚本
- 如果是 Python 项目，使用 `pytest`
- 捕获输出，解析结果

### Step 5：输出测试报告

```
## 测试报告

### 总体结果
- 通过：X
- 失败：Y
- 跳过：Z
- 通过率：N%

### 失败详情（如有）
| 测试 | 错误 | 文件 |
|------|------|------|

### 建议
- [需要补充测试的区域]
- [需要修复的代码问题]
```

### Step 6：失败处理

如果有测试失败：
1. 分析失败原因（测试写错 vs 代码有 bug）
2. 如果是测试写错 → 修正测试
3. 如果是代码 bug → 告知用户，建议修复方式
4. 重新运行测试验证
