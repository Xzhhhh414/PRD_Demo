# develop — 完整开发工作流

输入参数：$ARGUMENTS（PRD 文档路径或功能关键词）

---

## 执行流程

### Step 1：定位 PRD 文档

如果 `$ARGUMENTS` 是文件路径，直接使用。

如果 `$ARGUMENTS` 是关键词，搜索匹配的 `.md` 文件，列出结果让用户选择。

用 Read 工具完整读取 PRD 文档，提取关键需求点：
- 功能描述
- 模块清单
- 交互要求
- 约束条件

等待用户确认后继续。

### Step 2：获取 Figma 设计稿

通过 Figma MCP 工具读取设计稿：

1. 使用 `get_figma_data` 获取文件结构
2. 匹配与当前功能相关的页面/frame
3. 使用 `download_figma_images` 获取截图
4. 提取布局、样式、组件结构信息

如果没有设计稿或设计稿不适用，询问用户后跳过此步骤。

### Step 3：需求分析

使用 Agent 工具调用 Developer Agent 做分析：

```
subagent_type: "prd-workflow:Developer"
prompt: |
  请基于以下 PRD 和设计稿，给出实现方案。

  ## PRD 文档
  {粘贴 Step 1 提取的需求}

  ## 设计稿信息
  {粘贴 Step 2 提取的设计信息，如有}

  ## 现有代码上下文
  检查项目已有的代码结构、技术栈、现有组件和模块。

  请输出：
  1. 需要新建或修改的文件列表
  2. 每个文件的职责说明
  3. 组件/模块拆分方案
  4. 状态管理方案（如有前端）
  5. API 设计（如有后端）
  6. 需要确认的疑问点

  只做分析，不要写代码。
```

### Step 4：汇总疑问并确认

如果 Step 3 产出了疑问点，汇总后一次性询问用户。等待用户回答后，将答案补充到方案中。

### Step 5：生成开发需求文档

在 `dev-log/` 目录下创建文档。

**文件命名规则**：`[类型]-功能名-{YYYY-MM-DD-HHmm}.md`
- 类型：`feature`（新功能）/ `iteration`（迭代）/ `bugfix`（修复）
- 功能名：英文短横线连接
- 时间：当前时间，时分之间不用冒号（Windows 兼容）

**文档结构**：

```markdown
# [类型] 功能名

> 生成时间：YYYY-MM-DD HH:mm
> 来源：{PRD 文档路径}
> 设计稿：{Figma 链接 / 无}

## 1. 需求概述
{从 PRD 提取的核心需求描述}

## 2. 设计要点
{从 Figma 提取的界面设计要点}

## 3. 实现方案
### 3.1 文件变更清单
| 文件 | 操作 | 说明 |
|------|------|------|

### 3.2 模块设计
{组件/模块拆分、接口设计}

## 4. 待确认事项
{用户已回答的疑问及结论}

## 5. 测试清单
- [ ] 操作步骤 -> 预期结果

## 6. 实施记录
> 实施时间：{开始} - {结束}

### 6.1 实际变更文件
{实施完成后填写}

### 6.2 Code Review 发现并修复的问题
| 优先级 | 问题 | 处理 |
|--------|------|------|

### 6.3 已知限制
| 优先级 | 问题 | 说明 |
|--------|------|------|
```

告知用户需求文档已生成，询问："是否开始编码实施？"

等待用户确认后继续。

---

## 实施阶段

### Step 6：编码实施

使用 Agent 工具调用 Developer Agent 写代码：

```
subagent_type: "prd-workflow:Developer"
prompt: |
  请根据以下开发需求文档实现代码。

  {粘贴 Step 5 生成的文档内容}
  {粘贴 Step 2 的设计稿信息}

  实现要求：
  - UI 对齐 Figma 设计稿的布局、间距、颜色、字号
  - 功能对齐 PRD，不多不少
  - 先 Read 每个要修改的文件再 Edit
  - 新建文件用 Write
  - 完成后运行类型检查验证编译
```

### Step 7：Code Review

编码完成后，调用 Code Reviewer Agent：

```
subagent_type: "prd-workflow:Code Reviewer"
prompt: |
  请审查本次实施的所有代码变更。

  ## 需求文档
  {粘贴 Step 5 的文档内容}

  ## 变更文件清单
  {列出 Step 6 中实际新建和修改的所有文件路径}

  请 Read 每个文件的完整内容，按优先级审查并输出 PASS 或 FAIL。
```

### Step 8：修复循环

- **PASS** → 跳到 Step 9
- **FAIL** → 调用 Developer Agent 修复 🔴 问题，再次 Code Review，最多 3 轮

### Step 9：回填实施记录

更新 dev-log 文档第 6 章（实施记录）：
- 实际变更文件清单
- Code Review 发现并修复的问题
- 已知限制

### Step 10：运行测试

调用 Test Runner Agent：

```
subagent_type: "prd-workflow:Test Runner"
prompt: |
  请为本次实施生成并执行测试。

  ## 需求文档
  {粘贴 Step 5 的需求概述和测试清单}

  ## 变更文件清单
  {实际变更的文件列表}

  请：
  1. 分析项目测试框架
  2. 为变更的功能生成测试用例
  3. 执行测试
  4. 输出测试报告
```

### Step 11：告知用户并引导验证

输出实施总结：
- 变更文件清单
- Code Review 结果
- 测试报告
- 验证步骤建议

等待用户反馈。

### Step 11.1：用户反馈修复（循环）

用户反馈问题 → 调用 Developer Agent 修复 → Code Review → 更新 dev-log → 请用户重新验证。

### Step 12：自优化

回顾本次执行全过程，检查 4 类信号：

| 信号类别 | 来源 |
|---------|------|
| A. Code Review 重复模式 | Step 7 审查报告中的重复问题 |
| B. 用户修正 | Step 4、Step 11.1 中用户的纠正 |
| C. 实施摩擦 | 编码过程中 agent 遇到的困难 |
| D. 流程摩擦 | Skill 步骤执行中的跳过、卡住、重复 |

有优化提案时，逐条询问用户确认：
- **同意** → 修改对应 agent/skill 文件
- **拒绝** → 记录拒绝原因

在 `skills/develop/evolution.log` 追加进化记录。
