# 十周年活动 · 点券红包预算数值模型

> 本文档定义点券红包的多池数值体系，用于控制活动预算硬上限。
> 不涉及用户参与率预估，仅建立规则框架与理论天花板。
>
> **最后更新**：2026-03-11

---

## 一、设计原则

1. **多池独立**：根据不同场景设立不同额度的红包池，每个池有独立库存和独立预算
2. **消费分层**：剁手记录卡片按玩家消费金额分 5 档，不同档位对应不同红包池、不同保底和概率
3. **保底机制（假随机）**：消费达到一定门槛的玩家，保证获得对应次数的红包；保底次数内必定触发（100%），超出部分按概率判定
4. **通用兜底**：其他所有卡片共用同一个通用红包池
5. **库存即预算**：总预算 = Σ(各池库存 × 各池期望额度)，库存耗尽即停（含保底），硬性不超支
6. **空数据免疫**：卡片数据为空时仅领取 1 次纪念币，不参与红包判定；有数据的卡片每次领取均参与红包判定

## 三、通用卡片红包分析

通用红包（**RP-通用**，1~3 元，期望 2.00 元）覆盖除剁手记录外的所有卡片，每次领取 **20%** 概率触发。卡片数据为空时不触发。

单卡期望触发数 = **有数据用户数 × 人均领取次数 × 20%**


| 卡片       | 领取次数公式                            | 空数据可能   |
| -------- | --------------------------------- | ------- |
| 相伴时光     | min(10, max(1, 注册年数))             | 否（注册即有） |
| 夜行者      | min(10, max(1, ⌊深夜次数/50⌋))        | 是       |
| 新作预约     | min(10, max(1, ⌊预约数/10⌋))         | 是       |
| 冒险旅程     | min(10, max(1, ⌊游戏总数/50⌋))        | 是       |
| 游玩时光     | min(10, max(1, ⌊时长/200⌋))         | 是       |
| 喜爱类型     | min(10, max(1, 非零标签数))            | 是       |
| 游戏成就     | min(10, ⌊成就总数/50⌋ + 白金数)          | 是       |
| 我的挚爱     | 1（固定）                             | 是       |
| 独家宝藏     | min(10, max(1, ⌊独家数/10⌋))         | 是       |
| 编辑之选     | min(10, max(1, ⌊编辑推荐数/10⌋))       | 是       |
| 玩家之声     | min(10, ⌊评价条数/50⌋ + 嘴替数)          | 是       |
| 社区足迹     | min(10, max(1, ⌊发布数/100⌋))        | 是       |
| 社区点赞     | min(10, max(1, ⌊获赞数/500⌋))        | 是       |
| 深夜冲浪     | min(10, max(1, ⌊天数/15⌋))          | 是       |
| 徽章墙      | min(10, ⌊徽章总数/10⌋ + 黑金数)          | 是       |
| 同行伙伴     | min(10, max(1, ⌊好友/50⌋+⌊粉丝/500⌋)) | 是       |
| 游戏创作     | min(10, max(1, 作品数))              | 是       |
| Steam 卡片 | 1（固定）                             | 是（未绑定）  |
| 角色数据卡片   | 每角色 1 次                           | 是（未绑定）  |


> 精确计算需要各卡片的**有数据用户数**和**领取次数分布**（可从 feature-spec 中标注的 P50/P90/MAX 数据预筛获取）。
>
> 简化估算：假设活动参与用户 N 人，各卡片平均有数据比例 R，平均领取次数 C，则 **RP-通用期望触发 ≈ N × 17卡 × R × C × 20%**。
>
> 目前通过库存直接控制预算硬上限：**库存 × 2.00 元 = RP-通用预算上限**。

## 四、库存预算控制

通过设置各池库存来控制预算硬上限。**预算 = 库存 × 期望额度**，库存耗尽后该池停止发放（含保底）。

| 红包池 | 期望/个 | 理论需求(个) | 库存 | 预算(万) | 库存/需求 |
| --- | --- | --- | --- | --- | --- |
| RP-T1 | 1.50 元 | 1,337,716 | 100,000 | 15.0 | 7.5% |
| RP-T2 | 2.00 元 | 3,840,882 | 150,000 | 30.0 | 3.9% |
| RP-T3 | 4.50 元 | 673,844 | 40,000 | 18.0 | 5.9% |
| RP-T4 | 7.50 元 | 142,658 | 20,000 | 15.0 | 14.0% |
| RP-T5 | 15.00 元 | 28,025 | 5,000 | 7.5 | 17.8% |
| RP-通用 | 2.00 元 | — | 70,000 | 14.0 | — |
| **合计** | | | **385,000** | **99.5** | |

> 以上库存为示例配置（总预算 ~100 万）。调整方式：修改「库存」列，**总预算 = Σ 各池预算**。
>
> ⚠️ **关于 T2 覆盖率**：T2 理论需求 384 万个但示例库存仅 15 万，意味着仅约 4% 的 T2 用户能实际获得红包。库存耗尽后，后续 T2 用户（含保底资格用户）将无法获得该池红包。若需提升覆盖率，可增加 RP-T2 库存（每增加 10 万库存，预算增加 20 万）。
>
> 运营可在活动期间对任意单池追加库存，追加后即时生效。建议从保守配置起步，根据活动数据按需追加。
