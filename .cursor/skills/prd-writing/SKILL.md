## 工作流程

1. **先问再写** — 向用户确认产品背景、目标用户、要解决的问题、成功指标
2. **问题先行** — 文档开头清晰描述问题和背景，不直接列功能
3. **定义成功** — 每份文档包含可衡量的成功标准
4. **说明 Why Now** — 解释为什么现在做这件事
5. **保持精简** — 聚焦关键结果，避免冗余细节

## 写作风格

- 用「若…则…」的条件式描述分支逻辑
- 状态与规则优先用**表格**呈现，提高可扫读性
- 同一模块内的要点用列表组织，步骤用有序列表
- 默认使用**中文**撰写

## 内容边界

- **不包含测试数据和测试控件** — 文档只描述正式的产品功能和交互，不涉及 demo 测试入口、测试工具栏、测试按钮、调试参数等仅供开发/测试使用的内容
- **不包含技术方案** — 文档只用于表达交互体验和功能展示，不描述技术实现细节（如代码变量名、API 名称、数据结构、存储方式、路由写法、无障碍标签属性等）。若需说明跳转或深链行为，用产品语言描述（如「通过分享链接进入」），而非写出具体路由路径
- **不包含界面动效与视觉表现** — 动画效果（飞行动效、翻牌动画、过渡动画、脉冲反馈等）和视觉设计（配色、布局样式、图标风格等）以设计稿和 Demo 为准，文档不另行描述。文档只关注功能逻辑和交互规则
- **避免跨文档重复** — 若某功能模块已有单独文档维护（如数据回顾分享），则在本文档中不重复描述，将内容归入对应的独立文档。通用逻辑（如分享行为）应写在实际使用该功能的模块段落中，而非作为独立的全局章节
- **不包含界面具体文本** — 按钮文案、toast 提示语、标题副标题措辞、空态引导文案、宣传语等界面上显示的具体文本内容，以 Demo 和设计稿为准，文档中不逐字写出。文档使用功能性描述（如「toast 提示已领取」「显示空态引导文案」），而非引用具体文案（如「还没有获得奖品，快去抽奖和兑换吧~」）

## 写功能需求细案时

1. 先读模板：`.cursor/rules/demo-feature-spec-template.md`
2. 按模板结构逐模块展开：入口 → 正常流程 → 状态与规则 → 数据依赖 → 边界与异常
3. 最后补充模块间关系

## 从 demo 反推需求时

1. 通读 demo 代码，梳理功能模块和交互流程
2. 提取状态管理、数据结构、条件分支
3. 按功能细案模板组织输出
4. 先输出结构大纲让用户确认，再逐模块填充
